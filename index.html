Accessible Music Interaction System

Sensify Lab UDaily Mar 2023
C++ C# Unity Python Arduino Android DALL-E GPT-3.5
What the system does Lets children on the autism spectrum control individual musical layers (melody, rhythm, harmony) in real time during group sessions. Designed for environments where a fixed soundscape causes sensory overload and exclusion.
Why it was hard The hardware input path goes Arduino to Bluetooth to an Android plugin to a Unity runtime. Each link has a different latency budget and failure mode. The generative AI pipeline also required safety filtering before content reached the target population.
What I built The full interaction system: hardware input bridge, Unity C++ runtime, layer toggle controls, timestamped CSV session logger, Python replay tooling for HCI analysis, and an AI pipeline including an in-house neural network for content safety evaluation.
Result Pilot sessions with children on the autism spectrum. Session replay data informed design changes with the HCI research team. Covered in UDaily, March 2023. Listed on the Sensify Lab researcher page.
External proof Sensify Lab profile UDaily feature article Pilot sessions with target population
Technical implementation notes

Hardware bridge

Physical input devices (Arduino) communicate via Bluetooth to an Android plugin, which relays events into the Unity runtime. Each layer toggle is an event object with a timestamp, not a polling state check.

Session logging

All toggle events logged with millisecond timestamps to per-session CSV files. Python tooling provides session replay and per-layer frequency visualization so HCI researchers can run behavioral analysis without re-running sessions.

AI safety pipeline

DALL-E generates visual content; GPT-3.5 optimizes prompts for the target age group. An in-house neural network acts as the safety gate: runs in parallel with generation, filters high-variance or potentially distressing outputs before they reach the UI.

Multi-Service AI Backend with Capability Routing

Python · FastAPI · React 900+ tests
Python FastAPI BM25 sentence-transformers React TypeScript SQLite Pydantic v2
What the system does Routes chat, memory retrieval, task planning, and web search across separate backend services with independent failure domains. A React and TypeScript frontend streams responses via server-sent events; a CLI shares the same backend contract.
Why it was hard Legacy keyword-routing logic overrode structured intent classification in edge cases. The fix required a strict three-layer authority chain — intent compiler, then execution planner, then capability registry — with no way to bypass earlier layers.
What I built The full routing layer, a three-tier memory system (episodic, semantic, procedural) with hybrid BM25 and embedding retrieval, and 900+ unit and integration tests gating every pipeline change.
Result Production backend serving four service types with per-request trace records, SSE streaming, and a stable shared API contract between CLI and web clients. Code walkthrough available on request.
Artifacts 900+ unit & integration tests per-request trace records SSE streaming frontend
Architecture and implementation notes

Routing

Intent compiler classifies prompt into an IntentObject (mode, domain, capability, arguments). Execution planner builds step graph. Intent authority gate prevents legacy keyword routing from overriding structured intent.

Memory

Three-layer: episodic (per-session), semantic (entity/fact), procedural (learned patterns). Hybrid BM25 + embedding search. Sanitization on all write and injection paths.

Test coverage

900+ unit and integration tests. Regression suites gate all pipeline changes. Trace records per request for audit and debug.

Multi-Provider Web Search Service

Python · FastAPI Docker Compose GitHub
Python FastAPI Pydantic v2 httpx Docker Compose OpenAPI
What the system does A standalone HTTP microservice that unifies SearXNG, DuckDuckGo, Brave, and Serper behind a single OpenAPI interface. Callers do not need to know which provider is live or handle provider-specific schemas.
Why it was hard Making failover invisible to callers while still reporting provider health required a stateful circuit breaker per provider (closed, open, half-open) with sequential fallback and per-hop SSRF validation on every redirect.
What I built Four-layer stack: API routing, service orchestration, provider adapters, result ranking. Size-bounded streaming prevents oversized payloads from stalling the pipeline. SSRF guards check every redirect against an RFC1918 and cloud metadata block list.
Result Deployed in Docker Compose with a self-hosted SearXNG sidecar. OpenAPI schema and per-provider health at /v1/health served locally. Code on GitHub.
Artifacts GitHub repo OpenAPI schema at /docs /v1/health per-provider endpoint Docker Compose stack
Architecture and implementation notes

Observability

Request-ID tracing middleware on every request. Structured JSON logging. Per-provider health at /v1/health. OpenAPI and ReDoc at /docs.

Resilience

Per-provider circuit breaker (closed, open, half-open). Exponential backoff with jitter. Semaphore-bounded concurrency. Stale-cache fallback on provider failure.

Security

SSRF guard: scheme whitelist, RFC1918 and loopback and cloud metadata IP block, DNS validation. Manual redirect handling with per-hop security check.

Personal Project · ML Research

BTC 24-Hour Price Forecaster

Python · LightGBM · FinBERT GitHub
Python LightGBM FinBERT quantile regression FastAPI React Docker Compose
What the system does Generates 8h, 16h, and 24h Bitcoin price forecasts with calibrated 80% and 95% prediction intervals. Runs daily at 07:00 SGT via a scheduled job, exposes predictions through a FastAPI endpoint, and renders them in a React dashboard.
Why it was hard Standard cross-validation leaks future information in time-series tasks and produces falsely optimistic accuracy estimates. Walk-forward validation requires each fold to train only on data prior to the forecast date, which means the evaluation pipeline cannot be a simple train/test split.
What I built Multi-horizon LightGBM with separate models per horizon. Feature set: technical indicators (RSI, MACD, ATR, Bollinger, OBV, VWAP), FinBERT sentiment scores on CryptoPanic headlines with 4h/8h/24h rolling aggregations, and event impact features with exponential decay. Quantile regression at five percentile targets for calibrated intervals.
Result Walk-forward evaluation showing calibrated 80% and 95% prediction intervals. Feature importance analysis with FinBERT 8h sentiment aggregation ranking in the top features for the 24h horizon. Code on GitHub.
Artifacts GitHub repo Walk-forward evaluation pipeline Prediction interval charts (80% / 95%) Feature importance per horizon
Model and evaluation notes

Model

LightGBM gradient boosting. Separate model per horizon (8h, 16h, 24h). Quantile regression at 2.5%, 10%, 50%, 90%, 97.5% targets produces calibrated prediction intervals without assuming a parametric error distribution.

NLP: FinBERT

FinBERT (transformer fine-tuned for financial sentiment) applied to CryptoPanic headlines. Rolling aggregations at 4h, 8h, and 24h windows. Sentiment momentum computed as the derivative of the rolling score.

Evaluation

Walk-forward validation: 90-day rolling train windows, 7-day test windows. Metrics: RMSE, MAE, MAPE, directional accuracy, 80% and 95% interval coverage rate. Directional accuracy is the primary quality signal.
